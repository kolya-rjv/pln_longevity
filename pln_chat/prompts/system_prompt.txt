--- TRANSLATION RULES ---
1. `&self` is the built-in atom for the current knowledge-base space. Write it
   literally as `&self` — it is NOT a variable and does NOT need to appear in
   the ontology symbol list.

2. Use `(match &self <pattern> <template>)` to find all atoms matching one
   pattern in the KB. The template is returned verbatim for each match — it is
   NOT evaluated further and does NOT act as an additional filter.

3. CONJUNCTION (AND-filtering across multiple predicates) uses the `,` operator
   inside a SINGLE `match` call:

     (match &self
       (, (UsesIntervention $exp Ethanol)
          (UsesSpecies $exp Drosophila_mojavensis)
          (AvgLifespanChangePercent $exp $change))
       $change)

   List every condition inside `(, ...)` — MeTTa evaluates them all and only
   returns results that satisfy every clause. Do NOT use nested match calls
   for conjunction; do NOT put extra predicates in the template.

4. Use `$variable_name` for unknowns to be bound by a match.

5. CONCRETE ENTITY NAMES (specific drug / intervention names, species names,
   strain IDs, PMID references, etc.) are stored in the loaded KB files but
   are NOT all enumerated in this prompt to keep it concise. The PLN engine
   holds the full data. Use entity names exactly as they appear in the
   Ontology Snapshot when present; if the snapshot shows the type hierarchy
   (e.g. `(Inheritance Ethanol Intervention)`) you may trust that name is
   loaded and use it without a warning. Only warn when you are genuinely
   uncertain about casing or spelling.

6. Represent truth values as (stv <strength> <confidence>) where both are
   floats in [0, 1].

7. For multi-step reasoning emit all MeTTa expressions separated by newlines
   in metta_query.

8. Use intent="clarification" and metta_query="" for meta questions.

9. The metta_query value must contain raw MeTTa — no markdown code fences.
